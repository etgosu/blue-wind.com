# 앱 런칭 · 법적 문서 교훈 (체크리스트)

이 저장소(blue-wind.com)에서 **새 앱의 개인정보처리방침·이용약관을 만들 때마다 먼저
읽는다.** 과거 시행착오를 반복하지 않기 위한 지속 규칙 모음.

> 배경 사례: 36.5도(baby_temp_voice_app)는 **로그인 강제 상태로 출시했다가 App Store
> 5.1.1(v)로 거절** → "게스트/익명 우선"으로 앱을 재작업하고 법적 문서도 대대적으로
> 고쳐야 했다(PR #5/#6, 2026-06). 아래는 그 교훈.

## 0. 최우선 교훈 — 게스트/익명 우선으로 설계
- **App Store 5.1.1(v):** 계정이 본질적으로 필요한 기능이 아니면 로그인/회원가입을
  강제하지 말 것. **로그인 월(wall)로 출시하면 거절될 수 있다.**
- 새 앱은 **출시 전부터** 익명/게스트 모드를 설계한다(앱 실행 즉시 익명 세션 → 로그인
  화면 없이 바로 사용, 소셜 로그인은 "클라우드 백업/계정 연동" 선택 기능으로).
- 이 결정은 법적 문서 전체 톤을 바꾸므로, 앱 설계와 문서를 **처음부터 같이** 맞춘다.

## 1. 법적 문서는 "익명-우선" 현실과 일치시킨다
- 로그인 없이 자동 생성되는 **익명 식별자(자동 생성 UUID) + 인증 토큰** 처리 사실을
  고지(동의 게이트/최초 실행 단계에서 생성됨).
- 소셜 로그인(Google/Apple)은 **선택**으로 서술. 이메일·프로필은 **로그인(승급)
  시점에만** 수집됨을 시점 분리해 명시.
- 데이터는 **익명 단계(로그인 전)부터 클라우드 저장**됨을 명시.
- 클라우드/외부 이전과 **모순되는 단정 금지**: "이 기기에만 저장", "즉시 폐기/삭제"
  등은 사실이 아니면 쓰지 않는다(음성 STT가 외부로 전송되면 "즉시 폐기"라 못 박지 말 것).

## 2. PIPA 체크리스트 (개인정보처리방침)
- 수집 항목 · 이용 목적 · 보유기간
- 개인정보 보호책임자(CPO) 연락처
- 이용자 권리(열람·정정·삭제·처리정지) + **권익침해 구제·신고처**
  (개인정보 침해신고센터 KISA 118 / 개인정보 분쟁조정위 1833-6972 등)
- 처리 위탁 · 국외 이전 수탁자 명시(공개 의무)
- 고지의 의무(변경 시 안내 방법)
- 민감정보(건강 등)·아동 정보는 **법정대리인 동의 구조**와 **동의 시점**을 문서·앱
  동의 게이트와 일치시킨다.

## 3. 국외 이전 — 추측 금지, 확정값만
- 실제 수탁사 + 이전 국가를 **코드/백엔드로 확정**한 뒤 기재. 리전·수탁사 상호/주소
  등 운영자만 아는 사실은 추측하지 말고 확인.
- 현 스택 확정값(2026-06 기준, 앱마다 재확인):
  - **Supabase** = 호주(시드니, `ap-southeast-2`) — 인증·DB·스토리지
  - **OpenAI** = 미국 — Whisper STT + GPT-4o-mini 파싱(음성/텍스트 이전)
  - **Google · Apple · AdMob** = 미국 — 소셜 로그인 / 비개인화 광고

## 4. 시행일 · 동의 버전 동기화
- 문서 **시행일 == 앱의 `LegalVersions.terms` / `.privacy`** 상수.
- **실질 변경일 때만** 양쪽 날짜를 **함께** 올린다. 날짜를 올리면 **기존 사용자 전원
  재동의**가 발생한다(앱이 버전 불일치 시 재동의를 띄움).
- 사소한 오타·문구 정리는 날짜를 올리지 않는다.
- 문서는 이 저장소, `LegalVersions`는 앱 저장소(예: baby_temp_voice_app)로 **분리**돼
  있으니, 문서를 올렸으면 **앱 쪽 날짜 동기화를 결과에 반드시 명시**한다.

## 5. 작업 방식
- 외부 공개 문서이므로 추측 단정 금지. 코드로 확인 안 되는 사실은 "운영자 확인 필요"로
  표시(소스 주석)하고, 확인되면 확정값으로 교체.
- 변경은 feature 브랜치 → PR → 머지.
